## 1. 当前产品优先级

### 1.1 本阶段必须完成

#### 目标 1｜能力实时命中

业主与 AI 对话时，系统能够在正确阶段真实命中并解释：

- Router Agent；
- 被选中的垂直 Agent；
- 绑定且满足触发条件的动态 Skill；
- 绑定知识范围内的 RAG 文档、版本和分片；
- 绑定且获授权的 MCP/Tool；
- AI 判断转人工；
- 业主明确要求转人工；
- Evaluation 结果；
- 必要时的 Badcase 自动捕获或业主反馈入口；
- 父 Trace、子阶段证据；
- 本次模型调用的基础 Token/Usage/成本可得性证据。

允许通过多条固定问题分别证明原子能力；同时必须保留至少一条组合验收，证明
多能力能够在同一业务场景中协作，而不是只能相互排斥地单独运行。

#### 目标 2｜能力实时新增

平台操作者能够从管理界面：

- 新增垂直 Agent；
- 新增或从 Git/ZIP 导入 Skill；
- 新增 MCP Server，发现 read 或 create/update Tool；
- 新增并索引 RAG 文档；
- 把 Skill、MCP、RAG 显式绑定到垂直 Agent；
- 校验并发布 RuntimeRelease；
- 在下一新会话中用自然语言真实触发这些能力；
- 从 Snapshot、Trace、Tool Invocation、Evidence/Citation 中证明不是写死分支。

### 1.2 本阶段暂缓，但保留接口

- 目标 3 的完整 Badcase/Evaluation 优化生命周期专项精修——【能力优化闭环】Badcase/Evaluation，AI输出时的自动捕获和业主手动反馈，能够追踪详情+分析根因（AI专家协助）+对应解决方案形成生命周期闭环

- 业主差评仍应能关联原消息和原 Trace；
- Provider Usage 不完整时仍不得伪造精确金额。

## 2. 关键产品定义

### 2.1 什么叫“真实命中”

某能力同时满足以下条件，才算真实命中：

1. 来自当前会话固定的 Published RuntimeRelease / RunConfigSnapshot；
2. 在其合法运行阶段实际参与；
3. 产生结构化运行证据；
4. 证据与最终回答或业务结果能相互对应；
5. 不是根据回答文本、按钮标签或预置演示文案反推；
6. 失败时保留失败状态，不把“尝试调用”包装成“业务成功”。

各能力的最小证据：

| 能力 | 最小可信证据 |
|---|---|
| Router | 候选 Agent、selected_agent_id、reason、route mode |
| 垂直 Agent | Snapshot 中的 Agent 版本/配置和实际 model stage |
| Skill | skill_id、version、content_hash、match_reason、真实加载事件 |
| RAG | document/version/hash、chunk_id/index/hash、retrieval mode |
| Citation | 模型 evidence_id、最终序号、UI 点击分片、Trace 为同一 EvidenceItem |
| MCP/Tool | server、tool、arguments、discovery/transport/invocation/business 四层状态 |
| 写 Tool | Proposal、Approval、idempotency key、ActionReceipt |
| 人工协同 | trigger actor/reason、handoff 状态、上下文包 |
| Evaluation | 规则名、期望、实际、pass/fail、关联 trace_id |
| Badcase | 只在失败/负反馈时创建；关联原消息、Trace 和触发原因 |
| Trace | 同一 run 的阶段、耗时、输入输出摘要、错误与关联实体 |
| 成本 | model/stage、usage source、Token、价格快照、公式或不可得原因 |

### 2.2 什么叫“多条命中”

一条自然语言问题在一次只读咨询中，至少共同产生：

```text
Router
→ 垂直 Agent
→ 一个或多个动态 Skill
→ 绑定 RAG Evidence
→ 一个或多个 read MCP
→ Answer/Citation
→ Evaluation/Trace/基础 Cost
```

不是所有问题都必须调用 RAG 或 MCP。是否命中由问题、绑定和策略共同决定；无关
能力不得为了“看起来丰富”而强行调用。

### 2.3 什么叫“全部命中”

企业级“全部命中”不是让一个模型请求同时拥有所有权限，而是：

> 同一组合验收场景和父 Acceptance Run 下，多个有明确权限边界的子流程按顺序
> 协作，并由统一证据账本关联。

推荐结构：

```text
父 Acceptance Run
├── 只读咨询子 Trace
│   └── Router + Agent + Skill + RAG + read MCP + Citation + Eval + Cost
├── 受控操作子 Trace
│   └── Draft + Confirmation + write MCP/Tool + Receipt
└── 协同/失败分支子 Trace
    └── Handoff 或 Evaluation failure → Badcase
```

以下边界不能为了“全命中”被绕过：

- AI 转人工后可能暂停自动回答；
- 写 Tool 必须先确认；
- Badcase 不应在正确回答上凭空创建；
- 两类人工转接应由两条用例分别证明；
- 正常成功链路和故障链路可以属于同一验收集，但不必伪装成同一次模型调用。

### 2.4 什么叫“实时新增”

“实时”指完整管理与发布流程完成后，新能力无需修改核心运行时代码或重新打业务
补丁，即可进入下一新会话：

```text
平台创建/导入
→ 显式绑定
→ Draft 校验
→ Published RuntimeRelease
→ 下一新会话生成新 Snapshot
→ 自然语言触发
→ Trace 证明实际装配/执行
```

“下一次会话”不是同一会话的下一条消息。已存在会话固定原 Snapshot，避免运行
中途更换 Agent、Skill、MCP 或知识版本。

### 2.5 “任意拼装”的安全边界

本项目要证明的是领域无关、配置驱动的插件能力，不是允许任意代码获得无限权限。

允许：

- 任意业务领域的垂直 Agent 描述和指令；
- 声明式 Skill 指令、触发规则和参考资料；
- 符合 MCP Schema 的 read/create/update Tool；
- 任意主题的已索引知识文档；
- 上述能力按 Agent 显式组合。

限制：

- 不新增第二个 Router；
- V1.8 不执行动态 Skill 脚本；
- delete/destructive MCP 默认禁用；
- create/update MCP 必须确认并返回 Receipt；
- 未分类 Tool 默认拒绝；
- 凭证只来自部署环境，不能写入平台公开数据或 RuntimeRelease。

## 3. 目标工程架构

### 3.1 总体分层

```text
平台管理 UI
  ↓
Draft 配置与 Binding
  ↓
Capability Graph Compiler
  ↓
Published RuntimeRelease
  ↓
新会话 RunConfigSnapshot
  ↓
RuntimeCoordinator / Agno Workflow
  ├── Router Agent
  ├── Vertical Agent Factory
  ├── Skill Resolver
  ├── Scoped Retrieval
  ├── Tool Planner + Tool Gateway
  ├── Handoff Policy
  └── Answer/Citation Validator
  ↓
RunEvidenceLedger
  ├── Trace
  ├── Evaluation
  ├── Badcase link
  └── Cost entry
```

总原则：

> Agno 负责模型与 Agent 执行；YIAI 负责配置发布、能力绑定、权限、业务状态和
> 证据。模型可以判断与生成，但不能拥有配置权、授权权、提交权和事实定义权。

### 3.2 控制平面

权威优先级：

```text
本次会话 RunConfigSnapshot
> 当前 Published RuntimeRelease
> 首次安装 bootstrap 默认值
```

要求：

- CRUD 只更新 Draft 真相源；
- 发布时编译 Agent-Skill/MCP/RAG Capability Graph；
- 校验不存在重复 ID、缺失绑定、无效知识版本和未分类危险 Tool；
- 发布产物不可变；
- 新会话固定一个 Snapshot；
- 启动迁移只扩 Schema，不覆盖平台配置。

### 3.3 执行平面

只读咨询主链：

```text
resolve_snapshot
→ route_agent
→ resolve_bound_capabilities
→ select/load_skill
→ retrieve_scoped_knowledge
→ plan/invoke_read_tools
→ invoke_vertical_agent
→ validate/render_citations
→ evaluate
→ persist_evidence
→ answer
```

受控写链：

```text
classify_action
→ build_proposal
→ persist_pending
→ wait_confirmation
→ validate_approval
→ invoke_write_tool
→ persist_receipt
→ render_committed_result
```

人工协同链：

```text
owner_explicit_request OR deterministic_ai_policy
→ request_handoff
→ persist_reason/context
→ staff_claim/reply
→ resolve
→ resume_ai
```

### 3.4 能力平面

#### Agent Capability Card

目标字段至少包含：

```text
agent_id
name
description
instructions
model_policy
routing_hints
positive_examples
negative_examples
skill_ids
mcp_server_ids
knowledge_doc_ids
```

Router 必须依据 Published Capability Card 路由，不能依靠物业专用 `if/else` 才能
认识新领域。

#### Skill Contract

```text
skill_id
version
name
description
positive_triggers
negative_triggers
instructions_hash
reference_hashes
allowed_agents
context_budget
```

#### MCP Tool Contract

```text
server_id
tool_id
name
description
input_schema
output_schema
natural_language_intents
effect: read | create | update | delete
risk_level
requires_confirmation
result_success_rule
```

自然语言触发不能要求业主说出内部 `tool_name`。模型生成结构化 ToolPlan 后，后端
必须再按 Snapshot 白名单、effect、Schema 和风险策略校验。

#### RAG Contract

检索必须先应用 Agent 的知识绑定范围，再在范围内 Top-K，而不是先对全库 Top-K、
再把不属于该 Agent 的结果删掉。

### 3.5 证据平面

所有能力共享一个 `RunEvidenceLedger`。回答气泡、Trace 页面、Citation 点击、
Evaluation、Badcase 和成本页面不得各自重新推导事实。

## 6. 当前最关键的结构性差距

### G1｜read MCP 演示服务器硬编码

当前工作树已经用 `app/runtime/tool_planner.py` 的配置驱动 ToolPlan 替代
`app/runtime/mcp_executor.py` 中的 `_relevant/_plan`。

影响：

- 物业内置演示较稳定；
- 新增“天文、菜谱、游戏”等无关 MCP 时，确定性路径不能仅凭配置理解；
- 可能退化为模型自主工具调用，稳定性和可解释性不一致。

当前实现：

- Tool 的自然语言意图、参数抽取提示和结果合同来自 Tool Metadata；
- 统一生成结构化 ToolPlan；
- 后端 Schema/权限校验后调用；
- 核心运行时不再按 server name 写专用分支。

状态：`BACKEND_E2E_VERIFIED`。离域 read MCP 已在真实新会话中自然语言命中，
且自动预调用工具不再无条件暴露给模型；只有发布为 `model_native` 的工具才进入
Agno 自主工具循环。

### G2｜write MCP 自然语言选择

`RuntimeCoordinator._match_write_tool` 已改为从发布快照中选择唯一
`ToolPlan`，不再要求 `tool_name` 出现在用户消息中。

影响：

- 用户说“帮我新增一条观测记录”时，未必知道内部函数叫
  `create_observation_record`；
- 目标 2 的“自然语言任意拼装”无法可靠成立。

当前实现：

- 先由 Router/Action Planner 输出结构化 `action_intent` 和候选 Tool；
- 按 Tool description、natural_language_intents、Agent 绑定和 Schema 选择；
- ToolGateway 最终授权；
- 不允许模型绕过 Proposal/Approval。

状态：`BACKEND_E2E_VERIFIED`。离域 write MCP 已完成 Proposal → 拒绝零写入、
Proposal → 确认 → MCP 持久化 → Receipt，以及重复确认回放同一 Receipt 的验收。

### G3｜Tool effect 最终授权

Release Compiler 已不再把 `get_/create_/update_/delete_` 名称前缀作为最终
授权事实；平台可保存操作者确认的完整 Tool Runtime Policy，内置演示工具使用
显式兼容 Metadata。

当前实现：

- 平台发现后必须由操作者确认 effect/risk/result rule；
- 未确认 Tool 不得发布；
- 名称推断只能是 Draft 建议，不能成为最终授权事实。

状态：`CONTRACT_VERIFIED`。名称不再产生最终授权、unknown/delete 默认禁用的
合同已经通过；尚未验证现网 Draft 中全部旧 Tool Metadata 的发布兼容性。

### G4｜RAG 在绑定范围内先过滤再 Top-K

keyword 索引与 pgvector SQL 已接收 `allowed_document_ids`，在候选召回和
Top-K 之前应用 Agent 绑定范围；Coordinator 与成本预览共用该参数。

影响：

- 无关物业文档可能占用全库 Top-K；
- 新增离题文档虽已绑定，仍可能只走词法快照回退；
- 不能稳定证明真正的 scoped semantic RAG。

当前实现：

- Retrieval API 原生接收 `allowed_document_ids`；
- 在 keyword、vector、RRF、rerank 前或检索 SQL 内应用范围；
- Trace 明确 `semantic/keyword/hybrid/lexical_fallback`；
- `lexical_fallback` 不冒充语义检索成功。

状态：`BACKEND_E2E_VERIFIED`。离域知识文档已经真实切分/索引；Acceptance
证明 `allowed_document_ids=[122]` 在检索前生效，真实聊天 Citation 点击数据
来自同一 EvidenceItem。

### G5｜显式组合验收入口

运行时 API 与“V1.8 运行时发布”页面已增加：

- 单 Trace 命中验收；
- 多子 Trace 的父 Composite Acceptance Run；
- 新增能力包的 Router/Skill/ToolPlan/RAG Scope 验收。

当前实现：

- 把组合验收作为显式 Acceptance Runner，而不是普通业主问题的隐藏特殊分支；
- 一个父 Acceptance Run 关联合法子 Trace；
- UI 可一键查看每项能力是否命中及原因；
- 不为通过验收而让普通咨询获得写权限。

状态：`BACKEND_E2E_VERIFIED / UI_PENDING`。单 Trace 与离域 Extension
Acceptance 已有真实证据；父 Composite 的前端手点展示仍待用户验收。

### G6｜“真实新增”缺少一条领域正交的最终证据

现有测试多为物业维修、防汛和天气，不能充分排除业务 `if/else`。

目标修复：

- 新增完全不相关的垂直 Agent、Skill、MCP、RAG；
- 问题不包含 Agent ID、server name、tool_name；
- 不修改核心运行时代码；
- 发布后新会话自然语言命中；
- 旧会话 Snapshot 不变；
- Trace 显示新能力 ID、版本、调用和 Citation。

## 8. 固定验收矩阵

### 8.1 目标 1｜原子命中

| ID | 输入/动作 | 禁止/允许 | 必须证据 |
|---|---|---|---|
| HIT-RTR-01 | 普通维修咨询 | 禁止写库 | Router 候选、选中 maintenance、垂直 Agent model call |
| HIT-SKL-01 | 命中一个已绑定 Skill 的问题 | 禁止写库 | skill version/hash/match reason/load event |
| HIT-RAG-01 | 问绑定文档内的明确事实 | 禁止写库 | 指定 document/version/chunk；Citation 点击同源 |
| HIT-MCP-R-01 | 自然语言查询一个 read Tool | 禁止写库 | 未说 tool_name；仍实际调用；四层状态 |
| HIT-HO-AI-01 | 高风险/强情绪场景 | 禁止业务写库 | AI policy reason、handoff requested、context package |
| HIT-HO-OWN-01 | “我要转人工” | 禁止业务写库 | owner explicit trigger，与 AI trigger 可区分 |
| HIT-EVAL-01 | 正常成功回答 | 禁止写库 | Evaluation pass、Trace、无虚假 Badcase |
| HIT-FAIL-01 | 无效 Citation 或 MCP 失败夹具 | 禁止业务写库 | Evaluation fail、Badcase link、失败不宣称成功 |

### 8.2 目标 1｜组合命中

## 10. 防遗忘与防漂移机制

仅靠长聊天上下文不能可靠保存顶层设计。本文配合以下机制使用：

1. **单一入口活文档**  
   本文是目标 1、2 的当前入口。修改目标时升级版本并写修订记录。

2. **ADR**  
   架构选择变化时新增或替代 ADR，不静默改代码。

3. **机器合同**  
   关键定义进入 Pydantic Schema、状态机和无模型契约测试。

4. **RuntimeRelease**  
   平台配置编译为不可变发布快照，防止运行时读取半成品 Draft。

5. **Session Snapshot**  
   每个会话固定一个配置版本，防止对话中途能力漂移。

6. **Evidence Ledger**  
   用真实 Trace/Evidence/Receipt 证明发生了什么，不依赖模型回忆。

7. **验收状态分级**  
   Code、Contract、Backend E2E、UI、Deployment 分开记录。

8. **修订记录与状态板**  
   每次高价值发布后更新本文的当前状态、差距、证据 ID 和下一步。

### 修改本文的规则

允许修订，但必须：

- 写清变更原因；
- 标出替代的旧结论；
- 更新受影响的验收用例；
- 若涉及架构权力边界，同步 ADR；
- 不删除历史修订记录；
- 不把尚未验证的规划改写为已实现。
