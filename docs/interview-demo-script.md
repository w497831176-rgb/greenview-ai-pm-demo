# YIAI物业 V1.2｜面试 Demo 脚本（10 分钟版）

> 结构：前 5 分钟走业务闭环主线；后 5 分钟讲解技术后台与产品取舍。  
> 原则：已实现的 8 项能力明确展示；未实现的多模态报修明确说「下一版」。  
> 模型分层：
> - `deepseek-v4-flash`：默认文本 Router 与常规垂直 Agent 主力模型；
> - `deepseek-v4-pro`：复杂文本任务、Badcase/Darwin 深度分析、A/B 对比；
> - `kimi-k2.6`：仅 V1.3 用于图片/视频理解，不作为默认聊天模型。

## 第二部分：5 分钟技术后台讲解

### 5:00–5:40 架构分层：为什么是 Router + 垂直 Agent

**画面**：平台管理 → Agent 管理 / Skill 管理 / 知识库管理 / Badcase 库。

**口播**：
> "后台设计没有炫技式多 Agent。物业场景一次对话通常只有一个主意图，所以用 Router 做意图分发，垂直 Agent 处理各自业务域。这样调试简单、成本低、失败可定位。"

**讲解要点**：
- Router 只负责选 Agent，不处理业务。
- 垂直 Agent 绑定 Skill、知识、工具。
- 多 Agent 协作只在跨域复杂场景才需要，V1.3 再考虑。

---

### 5:40–6:20 RAG 与混合检索

**画面**：平台管理 → 知识库管理 → 检索调试。

**口播**：
> "RAG 用的是 BM25 + 向量 + RRF 融合 + 重排。BM25 负责精确术语与数字命中；向量负责语义与同义改写；RRF 融合两套排序；Cross-Encoder 重排模型对 Top-K 再打分，取最优片段送入模型。"

**讲解要点**：
- 文档切分：`chunk_size=512`、`chunk_overlap=64`、`split_strategy="auto"`。
- Embedding：`BAAI/bge-small-zh-v1.5`。
- 引用回显让回答可审计。

---

### 6:20–7:00 Skill、MCP 与工具调用

**画面**：平台管理 → Skill 管理 → MCP Server 管理 → Agent 编辑页。

**口播**：
> "Skill 是带触发条件与指令模板的能力单元，持久化为 SKILL.md，frontmatter 支持 version、name、trigger_condition 等元数据。触发用关键词子串 + 二元组 Jaccard，解决词序反转。MCP Server 以 stdio 方式接入，Agent 编辑页可以把工具绑定到 Agent。"

**讲解要点**：
- Skill 触发是 V1.2 轻量方案，V1.3 引入意图置信度和回归测试。
- 当前 **无自动版本历史/回滚**，修改 SKILL.md 即覆盖。
- MCP 当前仅支持 stdio，真实演示一次天气/日历工具调用。

---

### 7:00–7:40 模型分层：Flash / Pro / Kimi

**画面**：平台管理 → 模型配置。

**口播**：
> "默认生产模型是 DeepSeek V4 Flash，负责 Router 和常规垂直 Agent；Pro 用于复杂文本、Badcase/Darwin 深度分析、以及 A/B 对比；读取配置接口已经脱敏，不返回 api_key。Kimi 在 V1.2 没有接入，计划 V1.3 专门做图片理解。"

**讲解要点**：
- 80% 常规问题 Flash 足够。
- Pro 用于验证 Flash 是否足够，不滥用。
- A/B 接口 `POST /api/model-configs/ab-test` 并发返回两条 response，但**不自动返回延迟和成本**，需人工记录。
- API Key 优先环境变量，接口永不返回密钥。

---

### 7:40–8:30 人机协同与状态持久化

**画面**：员工 Tab 工单列表；平台管理 Badcase 库。

**口播**：
> "AI 不是替代人。员工接管、工单状态机、Badcase 闭环，这三层兜底让系统可运营。所有状态持久化到数据库，刷新不丢。"

**讲解要点**：
- 工单状态机：待处理 → 处理中 → 已完成。
- 接管时暂停 AI 自动回复。
- Badcase 是持续优化入口。

---

### 8:30–9:30 诚实讲边界：多模态是 V1.3

**画面**：业主 Tab AI 助手（即使前端有图片上传按钮，也明确说明当前未跑通）。

**口播**：
> "V1.2 没有实现多模态报修闭环。如果前端有图片上传按钮，那只是 UI 占位。V1.3 的完整链路设计是：图片先经过不调用 LLM 的媒体网关，判断 has_image 与 MIME=image/* 后，送入 kimi-k2.6 做视觉理解，输出 vision_result；然后 Flash Router 接收用户文本 + vision_result 文本摘要，再统一路由到对应垂直 Agent，最后由维修 Agent 预填工单字段。"

**讲解要点**：
- Flash Router 不直接接收图片。
- 仅当消息带图片附件时才调用 Kimi。
- 图片压缩、分辨率上限、低置信度追问都是 V1.3 的成本与体验控制策略。
- 跨域多 Agent 协同：也是 V1.3。
- Skill 触发升级为意图置信度：也是 V1.3。

## 录制注意事项

1. **预热**：首次调用可能因 Agno Registry 初始化等待 60–90 秒，录制前先发一条消息预热。
2. **敏感信息**：不要展示 `.env`、模型配置中的 API Key、网络请求详情。
3. **诚实边界**：多模态报修必须明确说「V1.3 实现」，不能写成已实现；A/B 延迟与成本必须说明是人工记录。
4. **节奏**：每个关键操作后停顿 1–2 秒。
5. **分辨率**：1920×1080，浏览器全屏，建议 125% 缩放。
