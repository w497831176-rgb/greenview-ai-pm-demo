【项目目的】
这不是要建设生产物业系统。它是我面试企业级 AI 产品经理时，用来证明自己对 AI 技术栈“知其然也知其所以然”的演示项目。

我只要求稳定证明三件事：
1. Agent 路由、动态 Skill、RAG 引用、MCP/Tool、人机协同、Badcase/Evaluation、Trace、成本治理都能真实命中且可解释；

【当前最大问题】
系统不是缺少功能，而是 app/chat.py 已经把只读咨询、状态写入、RAG、MCP、Skill、Router、Trace、成本、Badcase、人机协同都塞进一个流式入口。过去大量补丁相互干扰，出现过：
- 仅咨询却进入工单流程；
- 模型声称创建工单但数据库没有；
- “确认提交”未命中确认语义；
- Skill 已绑定却未命中；
- RAG 候选、回答引用编号、点击分片不一致；
- MCP 的调用/业务结果/失败混成一个数字；
- Provider usage 不完整却出现看似精确的成本；
- 超长 Skill 撑爆上下文。

V1.7.1 已做了局部保护：
- 工单使用确定性草稿→确认→真实写库；
- 完整标签字段能进入草稿；“确认提交”能确认；
- 动态 Skill trigger 可以拆分、`孩子`可归一为`儿童`；
- 单 Skill 运行时注入最多 6000 字符。
但不要把这些局部修复误读成系统已收敛。

【强制工作顺序】
第一阶段只读诊断。不要写代码、不要部署、不要调用真实模型、不要跑 Playwright/浏览器自动化、不要启动全量测试。

先交付《V1.8 运行时收敛设计》，必须包含：

A. 当前权威地图
- 每一项配置的真相源：Agent、Skill、Agent-Skill 绑定、MCP/Tool、知识库、RAG 索引、模型价格、预算、工单状态、Trace；
- 代码默认值、启动迁移、数据库、环境变量之间谁优先；
- 是否存在启动时覆盖用户平台配置的风险。

B. 明确把运行时拆成三条不可抢权的路径
1) 只读技术栈咨询路径：Router → Agent → Skill → RAG → 只读 MCP → Answer → Trace/Cost；不创建工单。
2) 受控业务操作路径：意图收集 → 草稿 → 用户确认 → 后端真实写库 → Trace；模型不得自行宣称成功。
3) 动态扩展路径：新增 Agent → 绑定 Skill/MCP → 固定问题运行时验收 → Trace 证据。

C. 逐条列出当前 app/chat.py 内应保留、应抽出、应删除或应降级的逻辑；不能只说“重构”。

D. 制定 6～8 条固定验收用例：每条必须写明输入、是否允许写库、预期路由、Skill、MCP、RAG 文档/分片、Trace、成本字段、清理策略。

E. 制定一个“引用契约”：模型文本、最终 citations、UI 点击分片必须由同一结构化数据驱动。

F. 制定一个“成本契约”：按 model/stage/usage source/price snapshot/formula 展示；usage 不完整时如何显示。

G. 给出最小改造计划，按 P0/P1 排序，估算每项影响文件和回归风险。

【先反向澄清】
提交设计前，只提出一组最多 5 个、真正会影响架构选择的合并问题。若无关键阻塞，请明确写“无关键澄清，以下按此假设推进”。

【得到我确认后的实施约束】
