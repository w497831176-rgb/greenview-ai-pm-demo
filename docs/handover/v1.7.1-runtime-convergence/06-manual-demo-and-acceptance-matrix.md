# 06｜人工演示与低成本验收矩阵

## 使用原则

- 用户负责前端点击和截图；接手 AI 负责代码/后端定向验证。
- 每次只验证一条黄金路径；不要用一条超长问题同时给所有技术栈施压。
- 发现问题时记录：输入、会话是否新建、页面实际输出、底部结构化证据、Trace ID、工单/数据是否真的变化。

## A. 只读技术栈路径

### 输入

```text
我是 3-2-1201 的王先生。卫生间天花板持续滴水。
本轮仅咨询，绝不创建、生成或提交任何工单。
请查询上海天气；查询我最近维修工单和待处理数量；
依据《常见维修问题 FAQ》和《物业维修服务承诺》说明应急处理和紧急维修时效。
每个关键结论给出可点击引用；最后汇总工具与实际引用文档。
```

### 必看点

- 路由：维修 Agent；
- Skill：维修工单处理（若其触发条件匹配）；
- 工具：天气、我的工单、我的未关闭工单、全小区脱敏聚合；
- 工具状态：调用成功与“业务结果 0”必须分别看，不能只看一次总数；
- RAG：FAQ 与服务承诺最终 citation；点击卡片查看的分片必须支持回答中的具体结论；
- 工单：本轮不应出现任何新 `WO-...`；
- Trace：路由、RAG、MCP、final response；
- 成本：Router 若缺 usage split 显示不可得是正确的，vertical 若有完整 token 才显示公式。

## B. 真实工单两阶段路径

### 输入序列

1. `厨房持续漏水，我要报修。`
2. 按页面要求补齐字段，或粘贴：

```text
房号：3-2-1201
问题描述：厨房持续漏水
紧急程度：紧急
联系电话：测试手机号
预约上门时间：明天上午
```

3. `确认提交`

### 必看点

- 第 1/2 轮：只能出现“待确认草稿”，不能声称正式创建；
- 第 3 轮：必须返回真实工单号；
- 到“我的工单”刷新，检查 ID、房号、描述、紧急度、预约时间是否一致；
- Trace 应明确是 `work_order_workflow`，模型 Token 为 0 / 未调用模型；
- 若模型话术说成功但列表没有工单，判定为 P0 失败。

## C. 动态 Agent + Skill 路径

### 平台操作

1. 创建/确认 `儿童教育Agent` 为 vertical 且启用；
2. 绑定 `zhangxuefeng-perspective` 或“孩子托管服务”；
3. 将触发词明确配置为多条：`儿童教育、孩子学习、不爱学习、学习动力、张雪峰`；
4. 新建业主会话。

### 输入

`我家孩子不爱学习怎么办？`

### 必看点

- 路由到儿童教育 Agent；
- 底部必须显示命中 Skill、命中原因；
- 没有知识库/课程工具时，不能虚构具体课程、收费、电话；
- Trace 的 Agent、Skill 与配置页绑定一致；
- 成本只属于实际使用的 Flash 调用。

## D. 动态 MCP 路径（只读）

### 输入

`请查询上海当前天气，再告诉我外墙渗水报修前要准备什么。`

### 必看点

- 维修 Agent；
- `get_current_weather` / `get_weather_advice` 实际调用；
- 天气演示 fixture 必须标注“演示数据，非实时公网天气”；
- RAG 没有相关文档时应承认未知，不得编造物业条款；
- 工单不会自动创建。

## E. 成本治理路径

1. 平台 → 成本治理；
2. 进入刚才对话的 Trace；
3. 分别查看 router / vertical / tool / RAG span；
4. 查看模型单价、价格生效日期、usage source、计算公式或不可得说明；
5. 查看 COST-01～04 的优化策略卡。

验收的关键不是金额大小，而是“每个金额都能回溯到模型、Token、单价快照和公式；没有字段时不伪装精确”。
