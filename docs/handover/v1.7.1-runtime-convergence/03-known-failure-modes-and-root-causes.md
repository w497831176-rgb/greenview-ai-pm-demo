# 03｜已知失稳模式与根因分析

## 结论

存在三种“上下文爆炸”，但最难的问题不是单纯 Token 长，而是**控制权上下文爆炸**：谁有权决定路由、调用工具、写工单、生成引用、记录成本，在同一条链路里变得不够单一。

## 1. Coding 会话上下文爆炸

这个项目经历了从 Agno demo、绿景智服、YIAI物业、V1.2～V1.7 多轮演进。每轮都增加了新目标、紧急补丁、旧版本描述和验收口径。新的 Coding 会话若只看代码，不知道这些决策的前后关系，就会：

- 复活已经淘汰的默认 Agent 或迁移逻辑；
- 把用户“演示原型，不要生产上纲上线”的边界误读为“不需要真实运行”；
- 看到问题后继续叠加 if/else，而非收敛旧路径；
- 在测试、NAS、浏览器自动化上花大量时间，却没有保护核心业务链。

**是否建议新开会话：建议。**但新会话必须先读本交接包、只读诊断并输出设计；否则只是把混乱上下文换了一个窗口。

## 2. 业务模型 Prompt 上下文爆炸

一轮业主聊天可能同时携带：

- Router 指令与短历史；
- 垂直 Agent 指令；
- 命中 Skill 的 Prompt；
- RAG 最终证据；
- MCP 工具结果；
- 图片理解摘要；
- 人工协同/工作流状态；
- 输出格式与引用要求。

此前曾出现一个导入 Skill 自动塞入二十多千字符，普通聊天 Token 被推到 48k+，导致超时。V1.7.1 已把单 Skill 注入上限压到 6000 字符，但这只是保护措施，不是最终设计。

**正确原则：**每条路径只带解决当前目标所需的上下文。咨询路径不需要工单字段草稿；写操作路径不需要为了“展示技术栈”强拉多份 RAG；动态能力演示不需要加载不相关物业历史。

## 3. 控制权上下文爆炸：最重要的根因

| 用户看到的现象 | 真正原因 |
| --- | --- |
| “本轮仅咨询”却被整理成待建工单 | 自由回答与状态工作流抢同一个入口；意图否定、草稿残留、字段识别的优先级没有被明确建模 |
| 模型说“工单已创建”，工单列表没有 | 写操作曾依赖模型/Tool 口头结果，而不是唯一后端控制器的成功返回 |
| `确认提交`没有写库 | 确认语义只写了部分词表，模型又接管了回答 |
| Skill 已绑定却没有命中 | 管理端保存格式与运行时解析格式不一致；“一个长字符串”和“多个触发词”不是同一份数据契约 |
| 回答出现引用，但点击是无关分片 | 召回候选、模型自行编号、最终 citation 卡片三条数据链没有强制绑定 |
| MCP 成功次数/业务结果看不懂 | 调用、参数合法性、业务返回、审计状态被折叠成一个“成功”概念 |
| Router 成本为空、vertical 有金额 | Provider 对不同调用返回的 usage 完整度不同；展示层需要区分“已配置单价”和“本轮可计算成本” |

## 4. 近期 Git 轨迹说明了什么

近期提交连续涉及：工单控制、Agent runtime contract、RAG evidence、Skill runtime、人机协同、MCP contract、动态 Agent、Evaluation、Router SSE、Trace、成本与引用。它们都触碰核心聊天链路。

这不是“某个 AI 特别差”，而是典型的 Vibe Coding 集成风险：单项功能实现很快，跨模块状态与契约收敛很慢。任何一个模块的“补丁式成功”都可能改变另一个模块的上下文、SSE 字段或数据库状态。

## 5. 不该归咎于硬件的问题

- Flash / Pro 文本推理在云端；NAS 或 5070Ti 不会修复路由、引用、确认写库、Skill 绑定等语义问题。
- 本地硬件升级可以改善向量检索、镜像构建、图片处理、SQLite/容器资源竞争，但不是本轮失稳的主因。
- 因此顺序应该是：先运行时收敛，再考虑是否把推理/多模态/索引迁移到 Windows 5070Ti。

## 6. 当前的必要止损

1. 冻结 `v1.7.1`，不继续在旧对话里“发现一个现象就补一个 if”。
2. 新会话先给出运行时权威地图和三条路径设计，得到确认后才动代码。
3. 每次版本只做一个完整高价值目标；删掉无价值自动化，不删业务验证。
4. 不再用“一个超级问题”作为所有模块的唯一验收。
