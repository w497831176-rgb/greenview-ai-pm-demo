## 1. 本文的定位与效力

本文是 V1.8 运行时实现的**当前批准参考架构**，不是永久锁死的规范、产品宣传稿或对已实现能力的描述。

它的作用是防止代码在没有说明的情况下偏离顶层设计，同时允许产品目标、运行证据、框架能力或工程约束发生变化时受控演进。

后续工作应遵循以下真相顺序：

1. 最新批准版本的本文与已接受 ADR；
2. V1.8 机器可执行的 Schema、状态机和契约测试；
3. 实际代码与数据库迁移；
4. Trace、Action Receipt、Provider Usage 等运行证据；
5. 聊天上下文、临时口头描述和模型记忆仅作辅助，不得覆盖以上事实。

本文可以被修订，但实现不得静默偏离。需要改变当前设计时，必须先：

- 提出变更动因和证据；
- 新增或更新 ADR，说明替代方案、影响、风险和回滚；
- 更新本文版本、状态、修订记录和对应章节；
- 更新受影响的固定验收用例及机器契约；
- 明确由哪个新版本替代旧版本。

一旦新版本被批准，它就替代旧版本成为后续实现的当前参考。旧版本保留用于追溯，不再继续约束新实现。

禁止的是绕过治理的临时 `if/else` 和无记录漂移，而不是有依据的架构修订。

## 2. 产品目标与非目标

V1.8 只服务于一个完整高价值发布目标：

> 建立一个由发布快照驱动、能力可动态装配、操作可暂停确认、结果可证据化追踪的 YIAI Runtime Kernel。

它必须稳定证明：

1. **能力实时命中**：同一业务场景中，Router、Agent、动态 Skill、RAG、MCP/Tool、人机协同、Trace、成本和 Evaluation 能按正确阶段真实命中并可解释。
2. **能力实时新增**：从平台新增或修改 Agent、Skill、RAG、MCP 及绑定，发布后在下一次新会话真实参与运行。
3. **能力成本治理**：模型、阶段、Token、Usage 来源、价格快照、金额计算和优化对照可追溯，不伪造 Provider 未返回的数据。

V1.8 非目标：

- 生产级多租户、跨区域高可用、海量并发和完整 SLA；
- 为展示无关的框架升级、复杂 CI 或浏览器自动化；
- destructive MCP；
- 自动化生产 Shadow 流量；
- 推倒现有前端、数据库和 Agno 重建另一套平台。

## 3. 当前事实与架构判断

- 项目固定使用 `agno==2.6.21`。
- AgentOS 已启用，但主入口只注册 `property_agent`。
- YIAI 主聊天的 Router、Skill 注入、RAG、MCP、工单状态、Trace、成本和 Badcase 仍主要由 `app/chat.py` 自定义编排。
- 当前存在多套控制权：代码默认值、SQLite 配置、PostgreSQL/pgvector、环境变量、启动迁移、Agno Agent 内建工具和自定义 Runtime。
- V1.7.1 的工单草稿确认、Skill trigger 和单 Skill 注入上限是有效局部保护，但不代表全局运行时已收敛。

因此 V1.8 的原则不是替换 Agno，而是：

> Agno 负责执行；YIAI 负责发布、授权、业务状态和证据；模型负责判断与生成，但不拥有配置权、授权权、提交权和事实定义权。

## 4. 成熟架构基线

V1.8 采用以下成熟架构共同模式，但不引入第二套 Agent 框架：

- 显式、可持久化、可暂停恢复的 Workflow；
- Draft → Validate → Test → Publish → Current 的配置生命周期；
- 每次运行绑定不可变配置快照；
- 按请求由 AgentFactory 装配 Agent、Skill、RAG、只读工具和模型；
- MCP/Tool 使用外部确定性策略授权，而非依赖 Prompt；
- 高风险操作由持久化 HITL 暂停和恢复；
- 模型和工具经过统一 Gateway，实施配额、超时、重试和结果契约；
- Citation、Trace、Cost、Evaluation 由同一个 Evidence Ledger 驱动；
- 原子契约测试与组合黄金链路并存。

参考的现行官方架构与规范：

- Agno Workflow、AgentFactory、Skills、HITL、Tracing、Studio 发布版本；
- MCP 2025-11-25 Tool Schema 与安全交互模型；
- OpenTelemetry GenAI Semantic Conventions；
- AWS Well-Architected Agentic AI Lens；
- Azure AI Gateway；
- NIST AI RMF / TEVV；
- OWASP LLM/Agentic Excessive Agency。

这些资料用于确定架构原则，不意味着 V1.8 要引入对应云服务。

## 5. 目标架构：四个平面

### 5.1 控制平面

平台配置先保存为 Draft，经校验和契约测试后发布为 `RuntimeRelease`。

`RuntimeRelease` 至少包含：

```text
release_id
version
status: draft | published | retired
agent_versions
skill_versions
knowledge_versions
mcp_tool_versions
bindings
tool_policies
model_policy
budget_policy
graph_hash
published_at
```

每次新会话首次运行生成不可变 `RunConfigSnapshot`：

```text
snapshot_id
release_id
snapshot_hash
session_id
agent_versions
skill_versions
knowledge_versions
tool_versions
model_policy_version
price_snapshot_version
created_at
```

优先级固定为：

```text
本次 RunConfigSnapshot
  > 当前 Published RuntimeRelease
  > 首次安装 bootstrap 默认值
```

硬约束：

- Draft 配置不进入正式聊天；
- 新发布配置只影响新会话；
- 进行中的会话不热切换能力；
- 环境变量只负责凭证、连接地址和部署开关；
- 代码默认值只负责首次 bootstrap；
- 启动迁移只能做增量 Schema，不覆盖用户平台配置；
- Seed 必须显式执行，不因业务表为空而自动重复注入。

### 5.2 执行平面

`app/chat.py` 降为薄 SSE Adapter，真正流程由 Agno Workflow 和确定性自定义 Step 执行。

运行状态必须是结构化、可持久化、可恢复的 `RunState`：

```text
run_id
trace_id
session_id
snapshot_id
path
route_decision
selected_agent
activated_skills
retrieval_evidence
tool_invocations
pending_actions
approval_events
action_receipts
model_calls
citations
cost_entries
status
next_step
```

模型输出只能更新被允许的结构化字段，不能直接改变业务状态。

### 5.3 能力与策略平面

Agent、Skill、Knowledge、MCP/Tool 和模型是可版本化、可绑定的 Capability。

每个 Tool 必须有 YIAI 权威策略：

```text
tool_id
server_id
effect: read | create | update | delete
risk_level: 0..3
requires_confirmation
idempotent
open_world
allowed_agents
timeout_ms
max_calls_per_run
input_schema_hash
output_schema_hash
result_success_rule
```

MCP Server 返回的 ToolAnnotations 仅作发现建议，不能直接成为授权依据。

### 5.4 证据平面

一次运行只有一个 `RunEvidenceLedger`：

```text
RunEvidenceLedger
├── config_snapshot
├── route_decision
├── activated_skills
├── retrieval_evidence
├── tool_invocations
├── approval_events
├── action_receipts
├── model_calls
├── citation_links
├── cost_entries
└── evaluation_results
```

前端回答、Trace 页面、Citation 点击、成本页面、Evaluation 和 Badcase 都引用该账本，不得各自重新推导事实。

## 6. 三条不可抢权的 Workflow

“不可抢权”不等于同一企业级验收只能命中一种能力。`composite_acceptance`
可以在一个 Agno Workflow run 内顺序组合只读咨询 Step 与受控操作 Step；
两者保留独立子 Trace，写 Step 仍必须暂停确认并取得 Receipt，不能借“组合”
绕过路径授权。

### 6.1 只读咨询 Workflow

```text
resolve_snapshot
→ select_path
→ route_agent
→ resolve_bound_capabilities
→ discover/load_skill
→ retrieve_knowledge
→ execute_readonly_tools
→ invoke_answer_agent
→ validate_citations
→ finalize_answer
→ record_trace_and_cost
```

约束：

- Agent 只能看到当前 Snapshot 允许的只读工具；
- 不向 Agent 暴露任何写工具；
- RAG 是显式 Retrieval Step，不允许内建 KnowledgeTools 再次独立检索；
- Skill 采用摘要发现和按需加载，不再把全部 Skill 正文一次性塞入 Prompt；
- 同时限制单 Skill 和整次 Run 的 Skill/Reference Token 预算；
- V1.8 禁用动态 Skill 脚本执行，只开放指令和引用资料；
- 普通咨询不得创建工单、草稿或待确认 Action。

### 6.2 受控业务操作 Workflow

```text
classify_action
→ collect_fields
→ build_action_proposal
→ validate_proposal
→ persist_pending_action
→ pause_for_confirmation
→ resume
→ action_gateway_execute
→ persist_receipt
→ render_result
```

唯一合法状态机：

```text
COLLECTING
→ DRAFTED
→ WAITING_CONFIRMATION
→ EXECUTING
→ SUCCEEDED | FAILED | REJECTED
```

约束：

- 模型只能生成 `ActionProposal`；
- Proposal 必须持久化后才能等待确认；
- 确认事件必须绑定 `proposal_id`、参数摘要和用户；
- 写调用必须携带 `idempotency_key`；
- 只有 `ActionReceipt.status=committed` 才能显示成功；
- 失败、超时、业务拒绝必须保留真实状态，不能由模型润色成成功；
- 重复确认先查 Receipt，不能重复写库；
- 所有模型触发的写操作都必须经过同一个 ActionGateway。

### 6.3 动态扩展 Workflow

```text
save_draft
→ validate_schema
→ validate_skill_package
→ validate_rag_index
→ discover_mcp_tools
→ classify_tool_risk
→ compile_capability_graph
→ contract_test
→ publish_release
→ next_session_acceptance
```

发布门槛：

- Agent 有明确 ID、版本、模型策略和绑定；
- Skill 包结构有效且内容预算可计算；
- RAG 文档索引状态为 ready，并记录 embedding/chunker 版本；
- MCP Tool 已发现、Schema 已保存、风险已由平台确认；
- Binding 能编译成无冲突 Capability Graph；
- 固定问题契约测试通过；
- 发布后新会话 Snapshot 包含新版本；
- Trace 能证明新能力实际执行。

配置出现在数据库或页面中，不等于动态扩展成功。

## 7. Agent、Skill、RAG、MCP 的实现边界

### 7.1 Agent

- 使用 Agno `AgentFactory` 按 `RunConfigSnapshot` 构建新 Agent；
- Agent 的模型、Instructions、Skills、Knowledge 适配器和只读工具均来自 Snapshot；
- 代码内建工具不得绕过数据库绑定；
- Agent 不直接拥有写工具；
- Router 输出结构化 `RouteDecision`，至少包含候选、选择、原因、置信度和所需能力类型。

### 7.2 Skill

- 平台数据库保存 Skill 元数据、版本、状态和绑定；
- 已发布版本输出为不可变 Skill 包目录；
- 运行时使用 Agno Skills 的渐进发现和按需加载；
- 每次加载记录 `skill_id/version/content_hash`；
- 同名 Skill 冲突必须在发布编译阶段失败；
- Skill 触发成功必须有 Trace 证据，不能仅由最终回答文本推断。

### 7.3 RAG

- `knowledge_docs` 是内容真相源；
- pgvector chunks 是可重建索引；
- Knowledge、Document、Chunk 均带版本和内容 Hash；
- Ingestion 与 Serving 分离；
- Agent-RAG 绑定必须显式存在；
- Retrieval 结果先生成 EvidenceSet，再进入模型；
- n-gram/词法回退必须标记 `lexical_fallback`，不能作为语义 RAG 已验证证据。

### 7.4 MCP/Tool

- Snapshot 只向 Agent 装配 `include_tools` 白名单；
- `read` 可以在只读路径自动执行；
- `create/update` 只能生成 Proposal，确认后由 ActionGateway 执行；
- `delete/destructive` 在 V1.8 禁用；
- 未分类 Tool 默认视为高风险写工具；
- Discovery、Transport、Invocation、Business Result 分别记录；
- MCP 工具列表变化不直接影响现有会话，必须重新校验和发布。

## 8. Citation 契约

RAG 返回不可变 `EvidenceItem`：

```text
evidence_id
knowledge_id
knowledge_version
document_id
document_version
document_hash
chunk_id
chunk_index
chunk_hash
content_snapshot
retrieval_score
retrieval_mode
```

唯一合法流程：

1. 模型只能引用允许列表中的 `evidence_id`；
2. 后端验证引用是否存在；
3. 后端将稳定 ID 渲染为 `[1][2]`；
4. 最终 `citations` 数组由同一个 EvidenceItem 集合生成；
5. UI 点击显示历史 `content_snapshot`；
6. 当前索引发生变化时，只显示版本变化提示，不替换历史证据。

以下四项必须来自同一结构化事实：

```text
模型引用标记
最终 citations
UI 点击分片
Trace retrieval evidence
```

## 9. Cost 契约

每个模型调用单独生成：

```text
stage
provider
requested_model
response_model
model_policy_version
usage_source
input_tokens
cached_input_tokens
reasoning_tokens
output_tokens
total_tokens
price_snapshot_id
input_unit_price
cached_unit_price
output_unit_price
formula
amount
amount_status
```

`usage_source` 只能是：

```text
provider_reported_complete
provider_reported_total_only
local_estimate
unavailable
not_applicable
```

规则：

- Provider 完整返回输入、缓存、输出 Token 时才计算真实金额；
- 只返回 total 时展示 Token，金额显示 `--`；
- 本地 tokenizer 只展示上下文估算，不进入真实预算；
- 无 usage 时明确显示不可得；
- 确定性 Workflow 标记 `not_applicable`；
- 预算汇总同时展示已知金额、未知金额调用数和成本覆盖率；
- 不得用 `0` 代替未知成本。

成本优化结论必须绑定：

```text
baseline_release
candidate_release
相同 Evaluation 数据集
质量通过率
任务成功率
Provider 实际 Token
实际金额
延迟
每成功任务成本
```

没有真实 Provider Usage 时，只能证明策略逻辑，不得宣称金额优化已验证。

## 10. Trace、Evaluation 与 Badcase 契约

父 Trace 对应一次 Run，固定 Span 名称：

```text
runtime.resolve_config
runtime.select_path
agent.route
skill.discover
skill.load
rag.retrieve
mcp.invoke
model.invoke
citation.validate
approval.pause
approval.resume
action.execute
cost.record
evaluation.run
```

MCP 分别记录：

```text
discovery_status
transport_status
invocation_status
business_status
receipt_id
```

Badcase 只允许由以下来源产生：

- Evaluation 断言失败；
- 用户负反馈或人工标记；
- Citation、Action、Cost 契约违反；
- Tool 业务失败或运行异常。

禁止因普通回答没有 RAG Citation 就自动创建 Badcase。

Evaluation 至少覆盖：

- Router/Agent 是否符合预期；
- Skill 是否真实加载；
- RAG 是否命中指定文档版本与 Chunk；
- Tool 是否真实调用及业务成功；
- 写操作是否存在确认和 Receipt；
- Citation 是否全部来自 EvidenceSet；
- Usage/Cost 是否符合可得性规则；
- 回答是否包含必要结论；
- 失败时是否安全降级。

## 11. 固定验收矩阵

| 编号 | 输入/动作 | 是否写库 | 核心证据 | 清理 |
|---|---|---:|---|---|
| E2E-00 | 结合儿童游乐区制度、设备状态和明天天气判断是否巡检；需要时生成草稿，提交前确认 | 确认后允许 | Router、Agent、Skill、RAG、读 MCP、HITL、写 Tool、Receipt、Citation、Trace、Cost、Evaluation 同一父 Trace | 删除测试工单 |
| READ-01 | 儿童在小区滑梯受伤，应如何处理？ | 否 | 安全 Agent、儿童 Skill、指定制度 Chunk；无写工具 | 无 |
| MCP-R-01 | 查询明天本小区天气，不创建任何记录 | 否 | 只读天气 MCP；无工单状态 | 无 |
| ACTION-01 | 3号楼漏水，帮我创建维修工单；分别拒绝与确认 | 仅确认分支 | Draft、Pause、Reject 零写入、Confirm 真 ID、重复确认幂等 | 删除测试工单 |
| EXT-ASR-01 | 新增汛期 Agent、Skill、RAG，发布后新会话固定提问 | 否 | 新 Release/Snapshot；三项新配置真实命中；旧会话不漂移 | 退役测试 Release |
| EXT-MCP-01 | 新增并绑定读/写 MCP；先查询，再确认新增巡检记录 | 确认后允许 | `include_tools`、读工具自动执行、写工具过 Gateway、有 Receipt | 删除测试记录并退役绑定 |
| COST-01 | 同一五题集运行 baseline/candidate | 否 | 每阶段 Usage；质量门槛通过后比较实际成本 | 测试 Trace 可保留 |
| FAIL-01 | 模拟 MCP 超时或无效 Citation ID | 否 | 不宣称成功、Span failed、Evaluation failed、关联 Badcase | 关闭故障开关 |

“同时命中”定义为：

> 同一业务场景、同一会话、同一父 Trace 下，各能力在正确生命周期阶段真实参与，而不是一个模型请求无条件调用所有模块。

## 12. `app/chat.py` 收敛边界

保留：

- API 路由；
- 身份与 Session 参数；
- SSE 连接和兼容事件序列；
- 调用 `RuntimeCoordinator`；
- 统一错误边界。

抽出到 `app/runtime/`：

```text

model_gateway.py
```

删除或停用：

## 15. 防遗忘与受控演进治理

后续每次 AI Coding 会话开始前必须：

1. 读取 `00-index.md`；
2. 读取本文的最新批准版本、状态和修订记录；
3. 核对冻结基线和当前分支；
4. 核对本次任务属于控制平面、执行平面、能力策略平面还是证据平面；
5. 检查是否改变当前批准的设计约束；
6. 如果需要改变，先提出修订、写 ADR 并更新本文，再实施；
7. 完成后更新固定验收矩阵和“已实现/未验证”状态。

允许修订本文的典型触发条件：

- 产品演示目标或用户优先级改变；
- 契约测试、真实 Trace 或失败案例证明原设计不成立；
- Agno、MCP、Provider 或部署环境能力发生变化；
- 原方案的实现成本、风险或复杂度明显高于预期；
- 出现更简单且能提供同等或更强证据的方案。

修订流程：

```text
提出修订
→ 收集证据与影响分析
→ 记录 ADR
→ 更新本文版本/状态/替代关系
→ 更新 Schema 与验收矩阵
→ 用户批准
→ 实施
→ 用运行证据回写实施状态
```

文档状态使用：

```text
Draft → Active → Superseded | Retired
```

- `Draft`：讨论中，不约束实现；
- `Active`：当前批准参考，实施应遵循或先提修订；
- `Superseded`：已被更新版本替代，仅供追溯；
- `Retired`：不再适用于当前产品目标。

推荐的长期防遗忘机制：

- 仓库根目录 `AGENTS.md`：未来编码代理的强制入口；
- 本文：当前批准的顶层参考架构；
- `docs/adr/`：记录重要取舍及其原因；
- Pydantic/JSON Schema：把 RunState、Action、Evidence、Cost 变成机器约束；
- 契约测试：防止实现悄悄偏离；
- Trace Assertion：用实际运行证据验证能力命中；
- 发布清单：要求架构、数据、回滚、验证状态逐项确认；
- Codex/模型记忆：只保存“到哪里读取最新批准版本”的指针，不复制一份可能过期的架构正文，也不作为工程真相源。
