# 01｜产品目标与演示契约

## 1. 项目不是“物业聊天机器人”

YIAI物业是 AI 产品经理的可运行面试作品。物业是载体，真正被面试官检验的是候选人是否理解企业 AI 系统如何：

- 用 Router / Agent 将问题交给正确责任主体；
- 用 Skill 把可复用业务做法按条件注入；
- 用 RAG 给出可核验的业务依据；
- 用 MCP / Tool 查询事实或执行受控动作；
- 让人工在需要承担责任时接管；
- 把错误、评估、Trace、成本变成可运营闭环；
- 允许平台通过配置扩展领域能力，而非每加一个场景就改死代码。

因此，面试中可以说“这是可运行的演示原型”，但不能声称已证明真实生产 SLA、海量并发、完整安全合规、自动自主修复或未真实接入的能力。

## 2. 三条必须可演示的产品主张

### 主张 A：常见技术栈能在运行时形成证据链

一次只读咨询可以展示：

`用户问题 → Router/Agent → 匹配 Skill → RAG 证据 → 只读 MCP 查询 → 业务回答 → citation/tool/trace/cost`。

这里“展示”不等于把所有文字堆进回答。每个环节必须有结构化证据：当前 Agent、命中 Skill、工具名/参数/结果状态、文档和分片、Trace 与成本来源。

### 主张 B：能力可动态新增

“动态”最低应满足以下闭环：

1. 平台创建或编辑一个垂直 Agent；
2. 为它绑定一个有明确正向/负向触发条件的 Skill；
3. 可选绑定受控 MCP Server/Tool；
4. Router 在下一次请求时可把合适问题路由给它；
5. Trace 显示路由理由、命中 Skill 和实际工具；
6. 管理页能看到配置与实际运行证据的对应关系。

“任意新增能力”不等于任意文本都必然有高质量答案。它的准确边界是：**在 Agent 职责、Skill 触发、知识来源和工具权限配置完整时，新领域能力可立即参与路由与执行。**

### 主张 C：成本治理不是一个“金额卡片”

成本链路应能回答：

- 本轮哪个阶段调用了哪个模型；
- Provider 实际返回了哪些 Token 字段；
- 单价来自哪个价格快照；
- 若 Provider 未拆分缓存/未缓存输入，为何不做假精确成本估算；
- Flash 是业主常规主力，Pro 仅用于 A/B、Darwin 等高价值深度任务；
- RAG Top-K、Skill 注入预算、工具结果摘要、历史轮数等怎样降低上下文与成本；
- 日/月预算达到 80% 和 100% 时，哪些后台高成本任务提示或阻断，业主 Flash 聊天为何仍可用。

## 3. 不能再采用的演示方式

不要再把“全技术栈”误解为一条超级问题必须同时触发所有模块。下面这类一次性要求会把只读查询、状态写入、RAG、多工具、引用、成本和会话状态混在一起：

> 查询天气、查询工单、引用两份知识、判断是否建单、禁止建单、汇总工具、展示成本。

它适合作为压力/集成场景，不适合作为核心演示的唯一验收。核心演示应该把**咨询、执行、扩展**拆开，让每条链路只有一个清晰的控制权。

## 4. 演示承诺与不承诺

| 可真实说 | 不应说 |
| --- | --- |
| 运行时按数据库配置加载垂直 Agent、Skill 与 MCP 绑定 | 所有任意新增领域均已达到专家级答案质量 |
| 知识回答可携带文档/分片引用并在界面查看证据 | 每次模型自然语言中的引用编号都天然可信 |
| 读工具调用、结果和失败会进入 Trace | 所有外部工具都是真实公网实时服务 |
| 真实工作流在用户确认后才能写库 | 模型说“已创建”就等于完成真实业务写入 |
| 按 Provider 返回的 Token 与价格快照计算或明确不可得 | Provider 未提供字段时仍可给出精确账单 |
